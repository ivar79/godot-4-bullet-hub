# دستورالعمل‌های اختصاصی پروژه بقاء تراز بالا (Bullet-Heaven / Survivor Mobile Game)

این فایل حاوی چارچوب‌ها، قراردادها و قوانین معماری توسعه پروژه بازی بقاء موبایلی با موتور Godot 4 و زبان GDScript است. تمام عامل‌های هوشمند (AI Agents) موظف به رعایت بی‌قید و شرط این الگوها هستند.

---

## 🎯 قوانین طلایی معماری (Golden Architectural Rules)

### ۱. معماری مبتنی بر رویداد (Event-Driven decoupled architecture)
*   هیچ گرهی (เช่น Player) نباید به طور مستقیم هندل گره دیگر (مثل Enemy یا UI) را گرفته و توابع داخلی آن را فراخوانی کند.
*   تمامی ارتباطات و تبادل داده‌ها باید از طریق هاب مرکزی رویدادها یعنی `EventBus` (`res://autoload/event_bus.gd`) انجام شود تا جفت‌شدگی (Tight Coupling) به صفر برسد و فریم‌ریت روی پردازنده‌های ضعیف موبایل پایدار بماند.

### ۲. بهینه‌سازی حافظه و استخر شیء (Object Pooling for Mobile)
*   **بسیار مهم:** به علت سربار بالای ایجاد و پاکسازی زامبی‌ها روی اندروید، استفاده از تابع `queue_free` برای دشمنان اکیداً ممنوع است.
*   تمامی دشمنان باید از طریق کلاس سراسری `EnemyPool` (`res://autoload/enemy_pool.gd`) بازیابی، تنظیم مجدد (`reset_state`) و به استخر بازگردانده شوند (`return_to_pool`).

### ۳. ورودی و ترسیمات بومی موبایل (Mobile Touch Visuals)
*   ترسیمات جوی‌استیک لمسی تک‌انگشتی فقط باید روی گره‌های دو بعدی ارث‌بری شده از `CanvasItem` مانند `JoystickVisual` (`res://scenes/ui/joystick_visual.gd`) انجام شود. گره ریشه HUD از نوع `CanvasLayer` بوده و نباید منطق شبیه‌سازی ترسیمی (`_draw`) روی آن قرار گیرد.
*   برای شبیه‌سازی آسان لمس با موس در ادیتور دسکتاپ، مقدار `emulate_touch_from_mouse` در تنظیمات پروژه (`project.godot`) همواره باید `true` باشد.

### ۴. تفکیک داده از منطق (Data-Driven Resources)
*   مشخصات و ویژگی‌های پایه‌ای بازیکنان و سلاح‌ها نباید داخل اسکریپت‌ها هاردکد (Hardcode) شوند.
*   همواره باید از سیستم قدرتمند Resource سفارشی گودو (`CharacterData` و `WeaponData`) برای جابجایی داده‌ها بدون نیاز به لود کدهای اسکریپت استفاده شود.

## 📁 ساختار درختی دایرکتوری‌های پروژه (Directory Structure)

```text
├── autoload/
│   ├── event_bus.gd         # مدیریت سیگنال‌های بازی
│   ├── game_manager.gd      # کنترل چرخه بازی، امتیاز، زمان و تجربه
│   ├── save_manager.gd      # ذخیره‌سازی محلی و آفلاین بالاترین امتیازات
│   └── enemy_pool.gd        # بهینه‌ساز رم موبایل و بازیافت دشمنان
├── resources/
│   ├── character_data.gd    # منبع داده مشخصات قهرمان
│   └── weapon_data.gd       # منبع داده مشخصات اتو-وپون‌ها
├── scenes/
│   ├── levels/
│   │   ├── main_level.gd    # راه‌اندازی صحنه گیم‌پلی
│   │   └── main_level.tscn  # سرهم‌بندی المان‌ها
│   ├── player/
│   │   ├── player.gd        # فیزیک و دریافت ورودی‌های جوی‌استیک
│   │   └── player.tscn      # بدنه فیزیکی قهرمان و دوربین تعقیب‌کننده
│   ├── enemies/
│   │   ├── enemy_base.gd    # رفتار هوش مصنوعی زامبی‌ها و دمیج ضربه‌ای
│   │   └── enemy_base.tscn  # گره فیزیکی دشمن در قالب مستطیل رنگی
│   ├── weapons/
│   │   ├── weapon_base.gd   # مدیریت شلیک سلاح و هدف‌گیری نزدیک‌ترین زامبی
│   │   ├── weapon_base.tscn # درخت گره‌های سلاح و تایمر شلیک خودکار
│   │   ├── projectile.gd   # فیزیک گلوله جادویی و قابلیت نفوذپذیری
│   │   └── projectile.tscn # گره برخورد برخورد گلوله‌ها
│   ├── items/
│   │   ├── xp_gem.gd        # رفتار مگنتی و صعود الماس تجربه به سمت بازیکن
│   │   └── xp_gem.tscn      # لوز درخشان متبلور بر روی زمین
│   └── ui/
│       ├── hud.gd           # بروزرسانی نوار جان، تجربه، لول و رکوردهای زمانی
│       ├── hud.tscn         # رابط کاربری بالای صفحه به همراه گره JoystickVisual
│       ├── joystick_visual.gd# ترسیم برداری شیک جوی‌استیک لمسی
│       ├── level_up_panel.gd# پاز بومی بازی و گزینه‌های جذاب ارتقای توانایی‌ها
│       ├── level_up_panel.tscn# پنجره سه گزینه‌ای لول‌آپ
│       ├── game_over.gd     # ثبت رکوردها پس از شکست خوردن قهرمان
│       └── game_over.tscn   # صفحه باخت بازی با پس‌زمینه زرشکی
└── scripts/
    ├── mobile_input.gd      # آنالیزور حرکتی سوایپ تک انگشتی بومی موبایل
    └── enemy_spawner.gd     # سیستم اسپاونر دوره‌ای و موج‌محور دشمنان
```

## ✍️ هنجار کدهای اسکریپتی (Coding Standards)

۱. **کامنتهای فارسی:** همواره از توضیحات و کامنت‌های فارسی روان برای تشریح گام‌های حساس معماری (مانند پاترن پولینگ یا مکش مگنتی الماس‌ها) استفاده کنید.
۲. **انواع پویا و ایستا:** به خاطر کارایی عالی موتور گودو ۴ روی پردازنده‌های موبایل، تعریف دقیق تایپ متغیرها (`var is_active: bool = false`) به عنوان اولویت شناخته می‌شود.
۳. **بصری‌سازی بهینه:** در غیاب اسپرایت‌های گرافیکی سنگین، از کدهای ترسیمی کارآمد درون تابع `_draw` (دو دایره با ضریب شفافیت بالا برای دستگیره جوی‌استیک، مربع رنگی با حاشیه برای زامبی‌ها و لوزی رنگی دورگیری شده برای الماس تجربه) استفاده نمایید.
۴. **دکمه‌ها و کلیدها:** همواره نام‌گذاری تگ‌ها، ویژگی‌ها و کلیدها را با سبک استاندارد `snake_case` در فایل پروژه ثبت کنید.
